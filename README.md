# MESBG Probability Engine

An exact, auditable probability solver for Middle-earth Strategy Battle Game Duel and damage calculations.

The engine is deliberately **rules-agnostic at the boundary**. A caller must verify the current rule state, profiles, formations, modifiers and exceptions, then encode those facts in `combat-input-v1`. The engine performs the arithmetic; it does not decide whether a model legally receives a bonus or can physically Back Away.

## Why this exists

Repeated ad hoc scripts are difficult to audit and easy to invalidate when one input changes. This package provides:

- immutable JSON inputs;
- exact rational enumeration, not Monte Carlo sampling;
- individual Duel-die identity;
- ordered, visible reroll decisions;
- standard and split To Wound calculations;
- conditional and unconditional damage outputs;
- an input SHA-256 hash for reproducibility;
- regression tests for known MESBG edge cases.

## Install

```bash
python -m venv .venv
. .venv/bin/activate
python -m pip install .
```

No runtime dependencies are required beyond Python 3.11+.

## Run

```bash
mesbg-probability solve examples/galadhrim_three_rank_vs_infantry.json
```

or without installing:

```bash
PYTHONPATH=src python -m mesbg_probability solve \
  examples/galadhrim_three_rank_vs_infantry.json
```

Validate an input without retaining the full result:

```bash
mesbg-probability validate examples/galadhrim_three_rank_vs_infantry.json
```

## Test

```bash
PYTHONPATH=src python -m unittest discover -s tests -v
```

## Supported in `combat-input-v1`

### Duel

- any number of individually identified D6 Duel dice;
- a Fight value per die source, allowing support models to supply the side's highest Fight;
- per-die modifiers clamped to 1–6;
- raw-result overrides, including a two-handed weapon's natural 6 remaining 6;
- one automatic raw-value reroll per die;
- ordered pool rerolls, including banner-like “choose up to one eligible die” decisions;
- rerolling an engaged or supporting model's eligible die;
- explicit equal-Fight tie probability, including Elven weapon or automatic tie effects;
- one-sided Duel Might sensitivity through explicit decision pools.

### Wounds and damage

- the standard Strength 1–10 versus Defence 1–10 To Wound chart;
- explicit one-stage or split wound targets;
- modifiers applied to every stage of a split roll;
- ordinary modified `6+/X+` first stages—natural results are required only when the source rule explicitly says so;
- ordered reroll rules with each physical stage rerolled at most once;
- Might spent on the first stage carrying to the second stage;
- exact damage distributions, including fixed damage and D3-style distributions;
- multi-Wound targets;
- repeated Fate attempts until a wound is prevented or Fate is exhausted;
- one Fate success preventing the full damage event generated by one successful Strike;
- Prone and Trapped as a single non-stacking doubled-roll condition.

## Deliberate boundaries

The engine fails closed or requires derived input rather than interpreting game text. Version 0.1 does not automate:

- whether a formation can Back Away or Make Way;
- whether cavalry charge bonuses are active;
- contested Duel Might where both sides can react repeatedly;
- a finite Might pool shared across several separate wound rolls;
- target selection in Multiple Combats;
- Heroic Strike distributions;
- Brutal Power Attacks, magic, rider/mount allocation, or bespoke natural-roll triggers unless translated into supported generic inputs;
- tournament or tactical value.

Run separate sensitivity cases where an unresolved rule or decision changes the input.

## Input principles

1. Use `side_a` and `side_b`, not attacker and defender, for the Duel.
2. Preserve die identity when modifiers, Fight values, rerolls or Might eligibility differ.
3. State the reroll decision order whenever both sides have optional rerolls.
4. Encode branch-specific states after the Duel: number of Strikes, Prone, Trapped, Defence and Fate.
5. Keep evidence references and assumptions in `metadata`.
6. Re-run from the full input after any correction; do not patch a downstream percentage.

See `schemas/combat-input-v1.schema.json`, the examples, and `docs/EDGE_CASE_MATRIX.md`.

## Reproducible releases

For runtime use from another project, pin all of:

- release tag;
- commit hash;
- release archive SHA-256;
- input schema version;
- engine version.

Do not fetch moving files from `main` during an analysis.
